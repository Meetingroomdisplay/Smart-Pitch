01
Operating Modes

Three Modes of Operation

SmartPitch supports three operating modes, selectable from the GUI. Each mode determines how heating decisions are made.

■

OFF

System disabled. No heating decisions are made. All outputs remain off.

▲

STANDARD

Basic frost protection using forecast data and configurable thresholds. Also records sessions for SmartPitch learning.

⭐

SMARTPITCH

Fully intelligent mode. Learning-driven timing and temperatures with slope-based safety net. Gets smarter every session.

Standard Mode

Two trigger methods are available (configurable in Settings):

Time-Based Trigger

Looks at the minimum forecast temperature in the next X hours (default 40h). Turns heating ON when forecast minimum falls below the frost threshold.

Forecast Temperature Trigger

Averages forecast temperature over the next X days (default 2 days). Compares against the frost threshold using actual or feels-like temperature.

Both methods apply hysteresis: heating turns ON below the frost threshold and OFF above threshold + deadband. A minimum run time (default 30 minutes) prevents short cycling.

Standard Mode Records Sessions Too

STANDARD mode records every session outcome into the learning database. This is how SmartPitch builds its initial data set when first deployed.

SmartPitch Mode

The intelligent mode follows five core design principles:

  1. No fixed preheat window — start time is 100% learning-driven
  2. Always start gentle — begin at a low temperature and ramp based on slope feedback
  3. Learning decides timing AND temperatures — both when to start and what setpoint to use
  4. Efficiency improves over time — reduces temperatures when passing with good margins
  5. Builds from Standard mode data — initial learning can come from STANDARD sessions

SmartPitch Sub-Modes

Sub-modeDescription
MONITORWatching conditions — no heating needed. Either no frost risk, or not yet within the learning-recommended heating window.
HEATINGActive heating to meet an upcoming session deadline. Learning sets the setpoint; slope control adjusts in real-time.
MAINTAINHolding a gentle 14°C between consecutive sessions within 48 hours when frost risk persists.
02
Decision Tree

How Decisions Are Made

The full decision flow is evaluated every control cycle (~60 seconds). This is the logic that determines whether SmartPitch should heat, monitor, or maintain.

Is there a training session scheduled? | +-- NO --> MONITOR ("No training scheduled - system idle") | +-- YES --> Has the deadline already passed? | +-- YES --> Another session within 48h AND frost risk? | | | +-- YES --> MAINTAIN at 14°C (gentle holding temp) | | | +-- NO --> MONITOR ("Session complete") | +-- NO --> Is there a frost risk for the next session? | +-- NO --> Severe frost in 7-day look-ahead? | | | +-- YES, within worst-frost lead time | | --> HEATING (early start for future frost) | | | +-- YES, but not yet time | | --> MONITOR ("severe frost in X days") | | | +-- NO --> MONITOR ("No frost risk") | +-- YES --> Within learning-recommended lead time? | +-- YES --> HEATING (learning sets setpoint) | +-- NO --> MONITOR ("start in Xh")
03
Learning System

Self-Learning Intelligence

SmartPitch categorises weather conditions into bands and matches current forecasts against historical sessions to determine optimal heating strategies.

Weather Bands

Four weather dimensions are used to classify conditions:

Temperature

T<=-6  |  T-6..-4  |  T-4..-2  |  T-2..0  |  T0..2  |  T>2

Cloud Cover

C0..10  |  C10..30  |  C30..60  |  C60..85  |  C>85

Humidity

H<60  |  H60..80  |  H>80

Wind Speed

W0..2  |  W2..5  |  W5..8  |  W>8

12-Hour Forecast Window

When classifying conditions for an upcoming session, the system averages forecast data over the 12 hours before the training start time — not the full forecast range. This captures the weather the pitch will actually experience during the critical overnight period. For example, if training is at 10am Friday, bands are calculated from 10pm Thursday to 10am Friday.

Tiered Fallback System

When looking up historical performance, the system uses a 5-tier fallback. Each tier progressively relaxes the weather match until sufficient data is found:

The system tries TIER 1 first. If no sessions exist at that level, it falls to TIER 2, then TIER 3, and so on. A minimum of 3 sessions is required for learning to be used.

How Learning Affects Heating

When to Start (Lead Hours)

The recommended lead time is adjusted based on historical pass rate:

Pass RateStrategyLead Time Adjustment
≥ 90% + margin > 2hEnergy optimisationUses most efficient parameters, or reduces by up to 30% of margin
≥ 90%High confidenceUses learned values directly
70–90%Moderate confidenceAdds 0.5h buffer to average
50–70%Low confidenceAdds buffer equal to fail margin + 1h
< 50%Very poor historyAdds 3h buffer, conservative 22°C setpoint

Raw vs Adjusted Lead Hours

Raw Lead Hours

The straight average of how long successful sessions took. This is what the database stores per tier. E.g. avg_pass_lead_hours = 50.8h

Adjusted Lead Hours

What control logic actually uses after applying pass rate optimisation and soil temperature offsets. E.g. adjusted_lead_hours = 39.6h — saving ~11 hours of energy.

The GUI lookahead narrative uses the adjusted value to show when heating will start, ensuring it matches what the control logic will actually do.

Two Independent Learning Matches

The system runs the tiered lookup twice every control cycle:

Next Session Match

Based on forecast bands for the next training session. Determines when to start heating for the immediate deadline.

Worst Frost Match

Based on forecast bands for the worst upcoming session in a 7-day look-ahead. Allows early start if severe frost is coming, even if the next session is warm.

Heated Sessions

A session starts when heating activates with a training deadline. During the session, the system tracks setpoint extremes (initial, max, min, average), slope intervention, and starting soil temperature. At the deadline:

PASS with Margin

Success achieved before deadline. Margin = minutes early.

PASS at Deadline

Frost clear and soil OK exactly at the deadline. Margin = 0.

FAIL

Frost still present or soil still frozen at deadline.

No-Heat Sessions

When a training deadline passes without any heating, the system records a "no-heat" session. This is crucial learning data — it tells the system when heating was unnecessary.

  • heated = false distinguishes from heated sessions
  • Pass/fail determined by the frost camera: frost detected at deadline = FAIL; clear = PASS
  • All setpoint and energy fields are NULL
  • Deduplication prevents recording the same deadline twice

Pass/Fail Criteria

A session is PASS when both conditions are met at the deadline:

  1. Leaf frost free — no frost detected by the camera (leaf_frost === false)
  2. Soil not frozen — soil temperature above the frozen threshold

A FAIL occurs when either condition is not met.

05
Soil Temperature

Underground Intelligence

8cm Depth

Shallow soil sensor. Responds faster to heating and cooling. Primary indicator for surface frost risk.

25cm Depth

Deeper soil sensor. More stable and slow to change. Represents thermal mass available to the pitch.

With Learning Data

When historical data exists, the system compares current soil temperature to the average from past sessions in the same weather band:

soilDelta = currentSoilAvg - historicalAvgSoilAtStart

Warmer Soil (positive delta)

More thermal mass available. Reduces setpoint by 0.5°C per degree and lead time by 0.2h per degree — the pitch needs less heating.

Colder Soil (negative delta)

Less thermal mass. Increases setpoint and lead time proportionally — more aggressive heating is needed.

Without Learning Data (Fallback)

When fewer than 3 sessions exist:

  • Soil deficit = max(0, 4.0 - currentSoilAvg)
  • Flow temperature boost: +1°C per degree of deficit below 4°C
  • Lead time: 6h + 0.5h per degree of deficit
06
Heat Curve

Temperature Mapping

The heat curve determines the flow temperature setpoint based on outside temperature. It uses a 3-point piecewise linear curve with a "knee" point:

PointOutside TempFlow TempPurpose
Point 1 (Cold)-10°C35°CMaximum heating for extreme cold
Point K (Knee)-2°C20°CMid-point transition
Point 2 (Mild)4°C15°CMinimum heating for mild conditions

The system interpolates linearly between points. Below Point 1 it caps at 35°C; above Point 2 it uses 15°C.

Fallback Only

The heat curve is used when no learning data exists (fewer than 3 sessions in any tier) or when in STANDARD mode. In SMARTPITCH mode with sufficient learning data, the learned setpoint overrides the heat curve.

07
Safety Net

Slope Control

The safety net monitors pitch temperature during heating and intervenes when things go off-track. It is dormant by default — normal control comes from the heat curve and learning system.

Slope Calculation

Uses linear regression on soil temperature data over a configurable window (default 60 minutes):

  • Averages 8cm and 25cm soil temperatures
  • Only uses samples where heating is active
  • Requires minimum 3 data points
  • Result: slope in °C per hour

Intervention Rules

ConditionSlopeAction
Success AchievedAnyReduces setpoint to save energy (up to 1°C per interval)
Temperature Falling< -0.3°C/hIncreases setpoint (proportional to fall rate, max 1°C)
Rising Too Fast> +0.5°C/hReduces setpoint to save energy (30% of rise rate)
Stable-0.3 to +0.5°C/hNo adjustment needed

Success condition: Leaf frost is clear AND avg_field_8cm_c ≥ 5°C. Adjustments have a minimum 120-minute interval to prevent oscillation.

Temperature Bounds

12°C

Minimum flow temperature (absolute floor)

14°C

Gentle start temperature (used for MAINTAIN mode)

35°C

Maximum flow temperature (absolute ceiling)

09
Protection

Safety Systems

Building Protect

Protects the building's heating from cold water returning from the pitch circuit. Closes the actuator when primary return drops below 45°C. Re-opens when it rises above 50°C. When active, pitch heating is suspended regardless of frost risk.

Forecast Override

Compares actual measured temperature against the weather forecast. If the pitch is 5°C+ colder than predicted, uses actual temperature for frost risk calculations. Catches inaccurate forecasts.

Anti-Cycling Hysteresis

Multiple mechanisms prevent rapid on/off cycling:

  • Frost threshold deadband — heating turns off at threshold + deadband, not at threshold
  • Minimum heat minutes — heating must run for at least 30 minutes before it can turn off
  • Slope adjustment intervals — minimum 120 minutes between setpoint changes
  • Building protect gap — 5°C gap between close and open thresholds

Maintain Mode

Consecutive Session Efficiency

When a session deadline passes and another training session is within 48 hours with frost risk still present:

1
Deadline Passes
Current session recorded as PASS or FAIL
→
2
Check Next
Is another session < 48h away with frost risk?
→
3
Maintain
Hold pitch at gentle 14°C
→
4
Ramp Up
When next session window begins, increase as needed

Energy Savings

Maintaining a gentle 14°C uses significantly less energy than a full cool-down and reheat cycle. The pitch retains thermal mass, so the ramp-up for the next session is faster and cheaper.

If the next session is more than 48 hours away, the system returns to MONITOR and will restart heating based on learning recommendations when the time comes.

11
Look-Ahead

7-Day Frost Scanning

SmartPitch doesn't just look at the next session — it scans all training sessions in the next 7 days to identify the worst upcoming frost event.

What It Calculates

Worst Temperature at Deadline

The coldest deadline temperature across all upcoming sessions in the 7-day window.

Hours Until Worst

How long until the severe frost event occurs, driving the start time calculation.

Example

If the next session is warm (no frost risk) but Saturday's session shows -4.2°C:

  1. Identifies Saturday as "severe frost coming"
  2. Looks up learning data for Saturday's forecast bands (independent tiered match)
  3. Calculates when heating needs to start for Saturday's deadline
  4. If that start time is now or has passed, begins heating early

Why This Matters

This prevents the situation where the system waits until Friday to discover it should have started heating on Thursday. Severe frost events need long lead times — the 7-day scan ensures the system is always prepared.

12
Energy

Energy Calculation

Power (kW) = Flow Rate (L/s) x 4.186 x DeltaT (°C) Where: Flow Rate = Water flow through heating system (currently 44.1 L/s) 4.186 = Specific heat capacity of water (kJ/kg·K) DeltaT = Secondary flow temp - Secondary return temp

Energy is only calculated when heat_enabled = true. When heating is off, energy shows as 0 kW. Season-to-date costs are accumulated in the database and displayed on the dashboard.

Scenarios

Typical Operation Scenarios

Scenario 1: Cold Night, Training Tomorrow

Evening: System in MONITOR, frost risk detected. Learning finds 22 similar sessions at TIER1_EXACT, 95% pass rate. Adjusted lead = 4.9h (raw was 5.8h but good margins allow reduction). At 4:06am (9am deadline - 4.9h): HEATING begins at learned optimal setpoint. Slope monitors the rise. By 8am: frost clears, pitch ≥ 5°C. PASS recorded with 60-minute margin.

Scenario 2: Mild Week (No Frost Risk)

Forecast shows all temperatures above threshold. Lookahead: "ALL CLEAR". System stays in MONITOR. Deadlines pass without heating. No-heat sessions recorded — frost camera confirms PASS (no frost present). These results validate that skipping heating was correct.

Scenario 3: Warm Today, Severe Frost Saturday

Thursday 11am: Friday's session is 2.8°C (safe). But 7-day scan finds Saturday at -4.2°C. Worst-frost match (TIER4_1BAND): adjusted lead = 39.6h. Saturday 9am - 39.6h = Thursday ~5:24pm. Lookahead: "SmartPitch plans to start heating around Thu 5pm". At 5:24pm: HEATING begins. Friday's session also benefits from continuous heating.

Scenario 4: Forecast Wrong (Override Activates)

Forecast predicts 5°C overnight. Actual drops to -1°C (6°C colder than predicted). Forecast Override activates (threshold 5°C exceeded). System uses actual -1°C for risk calculation. Heating activates. Crisis averted despite inaccurate forecast.

Scenario 5: Consecutive Training Days

Monday 9am training with frost → HEATING → PASS. After deadline: Tuesday training 24h away with frost risk. System enters MAINTAIN at 14°C instead of shutting off. Tuesday morning: ramps up from warm base. Uses significantly less energy than a full cold-start reheat.

Scenario 6: New System (No Learning Data)

Zero historical sessions. Falls through all tiers to TIER5_GLOBAL — no data. Falls back to heat curve for setpoint. Soil deficit adds extra heating if soil is cold. Default 6h lead time. Sessions recorded in both modes. After 3+ sessions, TIER4_1BAND provides data. System progressively improves.

Scenario 7: Building Protect

During pitch heating, the building primary circuit is cold (low demand). Primary return drops below 45°C → Building Protect activates. Pitch heating pauses (actuator closes). Building warms up, return rises above 50°C. Building Protect releases — pitch heating resumes. Schematic shows orange indicator.

Troubleshooting

Common Issues

No heating when I expected it

  • Check frost threshold (may be too low)
  • Check temperature type — "Feels Like" is more conservative
  • Verify forecast data is current (Forecast tab > stale warnings)
  • Check mode is SMARTPITCH or STANDARD (not OFF)
  • Check Building Protect isn't active (Schematic tab)
  • If SmartPitch says "start in Xh" — it's waiting for the learning-recommended start time

Heating when not needed

  • Review frost threshold (may be set too high)
  • Check Forecast Override (actual temp may be triggering it)
  • "Feels Like" can trigger heating in windy but above-freezing conditions
  • Verify forecast accuracy on Forecast tab
  • Check deadband isn't too small (rapid cycling)

Lookahead shows wrong start time

  • The lookahead uses adjusted lead hours (after efficiency optimisation), not raw averages
  • If only a few sessions exist (TIER4/TIER5), lead hours may be unreliable
  • As more sessions are recorded, predictions improve
  • Check which tier is being used (Mode tab shows match tier and session count)

System not learning

  • Ensure sessions are being recorded (Data > SmartPitch Sessions tab)
  • Need at least 3 sessions in a weather band for learning to apply
  • Check that pass/fail results are recorded (frost camera must be working)
  • Verify STANDARD mode is recording sessions

Energy display shows 0 or wrong value

  • Energy only calculates when heat_enabled = true
  • Verify flow rate setting is correct (44.1 L/s)
  • Check secondary flow/return sensors
  • Delta-T of zero or negative = zero energy
